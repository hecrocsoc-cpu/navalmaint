01

Descripción del proyecto

Guardamar Talía
Salvamento Marítimo
Astilleros Armón · 2009
31,9 metros
MTU 4000 M70 · 2 × 1740 kW
Node.js + Express + Prisma 6
PostgreSQL · Railway
React 18 + Vite · Vercel
JWT · 24h · bcryptjs
8 tests · Jest + Supertest

NavalMaint cubre el hueco entre el software industrial de grandes navieras (SERTICA, AMOS, ManWinWin) y las apps de recreo, con un mercado estimado de 2.500M$ en 2024. Cualquier embarcación puede registrarse con sus propios equipos, plan de mantenimiento y stock de repuestos. El caso de uso base es el Guardamar Talía, cuyo plan de mantenimiento real sirve como seed de datos.

05

Reglas de negocio

Autenticación y sesión
RN-AUTH-01Todo acceso a recursos protegidos requiere un token JWT válido en el header Authorization: Bearer <token>.
RN-AUTH-02El token caduca a las 24 horas. No existe mecanismo de refresh token.
RN-AUTH-03El payload del token incluye id, email y role del usuario.
RN-AUTH-04El vesselId se devuelve en login, registro y /auth/me, y se almacena en el cliente para filtrar todas las vistas.
RN-AUTH-05La verificación de roles se realiza en el backend con middleware verifyRole. El frontend solo oculta opciones por UX; la seguridad real reside en el servidor.
Barcos y equipos
RN-VES-01Solo el ADMIN puede crear, editar o eliminar barcos y equipos.
RN-VES-02La matrícula de un barco es única en el sistema (restricción a nivel de base de datos).
RN-VES-03Un MECANICO solo puede ver el barco al que está asignado. El filtro se aplica en backend por vesselId.
RN-VES-04Un barco puede tener múltiples mecánicos asignados. La asignación la realiza el ADMIN desde /users.
RN-VES-05Un MECANICO puede consultar los equipos de su barco, pero no crear ni modificar ninguno.
Tareas y logs de mantenimiento
RN-TASK-01Solo el ADMIN puede crear, editar o eliminar tareas de mantenimiento.
RN-TASK-02Cada tarea pertenece a un equipo y tiene una frecuencia entre 11 valores: de DIARIA a BIANUAL.
RN-LOG-01Tanto ADMIN como MECANICO pueden registrar logs (tareas completadas e incidencias).
RN-LOG-02Solo el ADMIN puede eliminar logs. El historial del mecánico es inmutable para garantizar la trazabilidad del mantenimiento.
RN-LOG-03Los logs se consultan con include anidado task → equipment para filtrar por vesselId en historial y dashboard.
RN-LOG-04Un log registra: fecha, horas de motor (opcional), estado (OK / INCIDENCIA / PENDIENTE) y observaciones en texto libre.
Stock
RN-STK-01Tanto ADMIN como MECANICO pueden añadir items y actualizar cantidades. El mecánico gestiona el inventario del día a día del barco.
RN-STK-02Solo el ADMIN puede eliminar items del catálogo de stock.
RN-STK-03Cuando cantidad < minimo, el item aparece en GET /stock/alertas.
RN-STK-04Los items se clasifican en clase A (crítico) o B (estándar), según normativa de piezas de respeto para buques de trabajo.
Dashboard
RN-DASH-01Las tareas pendientes se calculan dinámicamente comparando los días desde el último log con la frecuencia asignada. Sin campos extra en base de datos.
RN-DASH-02En la versión actual se muestran tareas DIARIA y SEMANAL. Las frecuencias largas están en el backlog.
RN-DASH-03Un ADMIN ve tareas pendientes de todos los barcos. Un MECANICO solo ve las de su barco asignado.

Script de test Postman (guardar token)

var json = pm.response.json();
if (json.data && json.data.token) {
  pm.collectionVariables.set('token', json.data.token);
}
